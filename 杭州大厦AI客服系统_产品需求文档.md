## 一、项目概述

### 1.1 项目背景

杭州大厦以百货业态为主，顾客咨询内容种类多、涉及面广，常规性、重复性问题较为集中。在AI技术重塑产业格局的时代背景下，公司拟建设AI客服系统，引入成熟稳定的大模型人工智能技术，为顾客提供智能问答等数字化工具，减轻一线员工重复性工作，提高服务效率和客户体验。同时通过系统建设，沉淀顾客的咨询记录、高频问题、服务偏好等数据资产，为公司经营管理提供数据支撑。

### 1.2 核心目标

| 维度 | 目标 |
|------|------|
| 业务目标 | 完成知识库搭建、客服体系建设、24小时在线服务、**VIP高端会员一对一直属人工服务** |
| 技术目标 | 知识库常见问题覆盖率≥80%、直接且准确解决问题的准确率≥90% |
| 规模目标 | 首期上线1个机器人+20个人工客服坐席（**含专属VIP会员经理坐席**） |
| 并发目标 | 1个机器人实例可同时处理5000+并发咨询 |
| 会话量 | 1年不少于300万条会话量 |

### 1.3 关键约束

| 约束项 | 要求 |
|--------|------|
| 交付期限 | 2026年9月30日前完成全部交付 |
| 模型选型 | 千问3.6为主力模型（3.7 max不稳定，deepseek v5不适用业务场景） |
| 部署方式 | 云端部署，服务期一年 |
| 安全合规 | 等保二级（或以上），日志留存180天以上 |
| 免费维护 | 系统上线后一年 |
| 上线辅导 | 不少于10人天 |

### 1.4 沟通会确认的关键结论

| 序号 | 结论 | 来源 |
|------|------|------|
| 1 | 项目偏向实用功能而非营销亮点 | 沟通会 |
| 2 | AI嵌入小程序咨询入口，作为现有服务流程的补充，不替代人工 | 沟通会 |
| 3 | **普通用户负面情绪/投诉咨询时，直接提供电话客服号码和服务台地址，不提供在线聊天转接** | **V2变更** |
| 4 | **VIP客户首次提问即识别身份，优先分配给专属会员经理，经理独占服务** | **V2变更** |

| 6 | **专属会员经理使用小程序端工作台接收VIP用户提问并推送多模态内容** | **V2变更** |
| 7 | 知识库更新采用人工管理（技术实时检测冗余冲突成本过高） | 沟通会 |

| 10 | 特殊问题采用"补丁"方式单独创建问答对（非复杂相似问法配置） | 沟通会 |

| 13 | 不做工单系统和外呼功能 | 沟通会 |

### 1.5 用户角色定义（V2更新）

| 角色编号 | 角色名称 | 描述 | V2核心差异 |
|----------|----------|------|-----------|
| R-01 | **普通用户** | 非会员/普通会员用户 | AI自动问答 → 转人工时仅提供**电话号码+服务台地址**，**不提供在线聊天转接** |
| R-02 | **尊享VIP用户** | 高卡级会员（钻石卡/黑卡等），拥有专属会员经理 | 首次提问即识别VIP身份 → **专属坐席独占服务** → 忙线时可选排队或电话直拨 |

| R-04 | **普通坐席** | 为普通用户提供服务的客服人员（**仅承接普通用户留言后续处理**） | 不承接普通用户在线聊天，仅处理留言等非实时场景 |
| R-05 | **知识管理员** | 负责知识库创建、审核、发布、维护的人员 | 与V1保持一致 |

### 5.2 核心模块详细设计

#### 5.2.1 知识库管理模块（与V1完全一致）

**文档上传与解析流程**：
```
用户上传文档
    │
    ▼
文件类型校验（PDF/DOC/DOCX/EXCEL/TXT/MD）
    │
    ▼
文件大小校验（≤50MB）
    │
    ▼
文件解析
    ├── PDF → PDF解析器 → 提取文本/表格/图片
    ├── DOC/DOCX → Office解析器 → 提取文本/格式
    ├── EXCEL → 表格解析器 → 提取行列数据
    ├── TXT → 文本解析器 → 直接读取
    └── MD → Markdown解析器 → 提取结构化内容
    │
    ▼
文本分块（Chunking）
    ├── 按段落分割
    ├── 控制chunk大小（建议500-1000字）
    └── 保留上下文关联
    │
    ▼
向量化（Embedding）
    └── 使用Embedding模型将chunk转为向量
    │
    ▼
存入向量数据库
    └── 关联原始文档、分类、标签、元数据
    │
    ▼
生成初步问答对（可选）
    └── 大模型基于chunk生成Q&A对
    │
    ▼
提交审核（待审核状态）
```

**双人审核流程**：
```
知识条目创建（状态：待审核）
    │
    ▼
审核人A审核
    ├── 通过 → 状态：审核中(B)
    └── 驳回 → 状态：已驳回 → 通知创建人修改
    │
    ▼
审核人B审核
    ├── 通过 → 状态：已发布 → 同步至AI检索库
    └── 驳回 → 状态：已驳回 → 通知创建人修改
```

**关键设计决策（基于沟通会）**：
1. **知识冗余/冲突检测**：技术实时检测token消耗成本过高，采用人工管理方式
2. **旧版本处理**：下架旧版本 + 上传新版本的方式管理
3. **特殊问题处理**：采用"补丁"方式，为模型难以理解的特殊问题单独创建问答对映射
4. **过期管理**：设置有效期，到期自动下线，到期前预警提醒

## 十三、实现考虑与注意事项

### 13.1 普通用户与VIP用户的体验差异

- **普通用户**：AI自助服务为主，不提供在线人工聊天转接。需确保AI回答的准确性和多模态内容的丰富性，最大程度减少用户转人工需求。
- **VIP用户**：专属人工服务为核心。需确保专属经理响应及时，排队机制公平合理，电话直拨选项作为兜底方案。

### 13.2 专属经理排班与状态管理

- 专属经理的工作时间应与VIP用户活跃时段匹配
- 建议设置自动状态切换（如非工作时间自动离线）
- 经理离线时应自动触发排队+双选项逻辑
- 需考虑多个VIP用户对应同一经理时的并发处理能力

### 13.3 多模态内容数据对接

- 需提前与杭州大厦现有系统对接，确认活动/商户/商品/导航数据库的API接口
- 建议采用内容映射表（content_mapping）统一多模态内容格式
- 需考虑内容过期、下架等状态的同步机制
- 坐席推送的内容应支持预览，避免发送错误信息

### 13.4 小程序端推送实现方案

- **实时推送**：WebSocket长连接（坐席小程序在线时）
- **离线推送**：微信小程序订阅消息（坐席小程序离线时）
- 双通道保障：优先WebSocket，失败降级为订阅消息
- 需处理小程序前后台切换场景

### 13.5 VIP排队预估算法

```
预计等待时间 = 排队位置 × 平均服务时长

其中：
- 平均服务时长 = 该经理最近N个会话的平均时长（滑动窗口）
- 建议N=20，权重衰减：最近会话权重更高
- 每分钟更新一次预估时间
- 实际等待时间与预估偏差超过50%时触发告警
```

### 13.6 AI与人工服务的边界

- **普通用户**：全程AI服务，转人工仅提供电话/地址引导
- **VIP用户**：首次提问AI识别身份并路由，一旦接入专属经理，AI不再介入
- **VIP未接入经理前**：AI可正常回复用户问题，同时尝试连接经理
- 需在会话状态中标记AI是否介入，避免状态混乱

### 13.7 降级策略

| 场景 | 降级方案 |
|------|----------|
| 会员系统不可用 | 默认按普通用户处理，AI正常服务 |
| 专属经理信息缺失 | 降级为普通用户流程 |
| 专属经理全忙/全离线 | VIP排队+双选项 |
| 小程序推送失败 | 降级为短信通知 |
| 多模态内容API不可用 | 坐席端显示"内容暂不可用"，支持文字回复 |
| AI服务不可用 | 普通用户直接提供电话+地址；VIP用户直接排队/电话 |
